The 4 things you must have for it to be usable as a backbone

If GrooveGO is going to support an Anytype-like layer, it needs these explicitly:

1. Identity is not optional

Not just:

device IDs

But:

user identity
device identity
signing keys
rotation/revocation

Without that, you don’t have “Groove,” you have “Dropbox with merge.”

2. Membership must be first-class

Groove’s killer feature was:

“who is in this space?” is cryptographically enforced

So you need:

workspace membership list
signed membership changes
history of joins/leaves

Otherwise:

you can’t safely share anything
3. Data must be scoped to shared contexts

If everything is just:

a global CRDT log

You’ll hit a wall.

You need:

per-workspace logs
per-object encryption (optional but powerful)
clear boundaries of replication
4. Deterministic merge semantics

If you want an Anytype-style object graph on top:

links must not “half-break”
schema must not diverge
references must survive offline edits

That means:

stable IDs
conflict resolution rules that are predictable
Where GrooveGO likely already helps you

Go is actually a strong choice here:

easy concurrency → good for replication engine
networking primitives → useful for peer sync
portable → can run everywhere

So yes, it can absolutely become:

the “Groove-like kernel” layer

What you should NOT try to do

Don’t try to:

“embed Anytype into GrooveGO”

Anytype is not designed to be:

a thin client
or a pluggable frontend

You’ll fight it the entire way.

What you should do instead

Use GrooveGO as:

the source of truth + replication + trust layer

Then build a thin layer that exposes:

objects
relations
types

Basically:

GrooveGO (CRDT + identity + membership)
        ↓
Object Graph Layer (Anytype-like)
        ↓
UI (web / desktop)
The “aha” moment

If you get this right, you won’t just recreate Groove.

You’ll get something Groove never had:

a semantically rich shared knowledge graph with real trust guarantees

That’s actually a next-generation system, not a clone.

Honest assessment

Can GrooveGO be used?

✔ Yes — as a foundation

❌ No — not yet as a drop-in backbone for an Anytype-like system

If you want a concrete next step

Don’t add UI yet.

Add this first:

Workspace abstraction
create/join/leave
Signed membership list
Per-workspace log
Object ID layer (UUID + version)

Once you have that, suddenly:

an Anytype-style layer becomes straightforward

Where I’m going to push you (important)

Right now your plan is correct but underspecified in the hardest spots.

There are 3 places where this will either succeed or stall:

1) Identity: you need to choose your unit of truth

You wrote:

user vs device, rotation, revocation

But you need to decide something very concrete:

Is the system anchored on:
user identity (primary)
devices are delegates

or

device identity (primary)
users are emergent groupings

Groove picked:

user identity with device keys underneath

If you don’t lock this early:

membership logic becomes inconsistent
revocation becomes messy
invites become ambiguous

👉 My recommendation:
Start with device-first internally, but expose user identity externally

2) Membership: CRDT ACL is not trivial

This line is doing a lot of work:

signed CRDT ACL, content-addressed workspace IDs

You need to answer:

Is membership:
a log of events (join/leave/invite/revoke)?
or a state snapshot?
Who is allowed to:
add members?
remove members?
rotate keys?

Groove solved this with:

role-based authority
signed operations
replayable membership history

👉 If you skip history, you lose:

auditability
conflict resolution
offline correctness
3) Replication boundary (this one will bite you later)

You wrote:

per-workspace topic, per-object oplog

That’s good—but incomplete.

You must define:

Can an object exist in multiple workspaces?
Are object IDs:
globally unique?
or workspace-scoped?

Because this affects:

deduplication
sharing semantics
graph integrity

👉 Strong suggestion:

Start with:

objects are workspace-scoped

Global objects can come later—but they explode complexity.

The hidden missing piece

There’s one thing not explicitly in your plan that should be:

→ Encryption model

You mention identity + membership, but:

Are logs encrypted per workspace?
Are objects individually encrypted?
How do keys rotate when membership changes?

Groove’s real strength wasn’t just CRDTs—it was:

encryption aligned with membership

If you don’t define this early, you’ll have to redesign your storage later.
